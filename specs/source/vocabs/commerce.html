Payee

An entity, including a destination account, that receives monetary funds during a Transaction.

Status
stable
Parent Class
owl:Thing
Properties
com:rate, com:rateType, com:currency, com:destination, rdfs:comment, com:rateContext

The following example expresses a flat payment of $0.05 USD. 

{
   "a": "com:Payee",
   "com:rate": "0.05",
   "com:rateType": "<com:FlatAmount>",
   "com:currency": "USD",
   "com:destination": "<http://dev.payswarm.com/person/john-doe#account-3>",
   "rdfs:comment": "Description about payment reason",
}

PayeeRule

A Payee Rule is a mechanism that can describe limits and allowances on the payees during a transaction. These rules are typically used to apply algorithmic formulas to the final price of a particular good. For example, minimum and maximum prices, tax rates, when taxes are applied, and preventing additional payees are a few examples of the types of payee rules that can be specified using PayeeRules.

Status
stable
Parent Class
owl:Thing
Properties
com:maximumRate, com:minimumRate, com:destinationOwnerType, com:rateType, com:limitation

The example below expresses a maximum payment rate of 10% to go to a PaySwarm Authority. This limits the amount of profit a PaySwarm Authority can make to 10% of the final sale price. 

{
    "a": "<com:PayeeRule>",
    "com:maximumRate": "10",
    "com:rateType": "<com:InclusivePercentage>",
    "com:destinationOwnerType" : "<ps:Authority>"
}

The following example expresses that all rateTypes must be either flat rates or exclusive percentages. That is, inclusive percentages are not allowed.

Why would anyone ever need this? We need to explain.

{
    "a": "<com:PayeeRule>",
    "com:rateType": ["com:FlatAmount", "com:ExclusivePercentage"]
}

The example below specifies that no additional payees are allowed to be added to the list of payees for a transaction. 

{
    "a": "com:PayeeRule",
    "com:limitation": "ps:NoAdditionalPayees"
}

stable
Parent Class
owl:Thing
Properties

The example below shows a transaction processor can transfering funds from a local financial account to an external financial account.

   "com:forTransaction": "";
   "com:source": "",
   "com:destination": "",
   "com:amount": "8.00",
   "com:currency": "USD",
   "rdfs:comment": "Transfer of $8.00 on 2010-10-16 at 12:34pm - Paying Jane back for pizza last Monday"
}

Transaction (was ps:Transaction)

A record of a commercial exchange between two entities resulting in the transfer of monetary funds from one or more source accounts to one or more destination accounts.

Status
stable
Parent Class
owl:Thing
Properties
com:transfer, com:amount, com:currency, rdfs:comment, com:date, ps:originatingClient

The example below describes a commercial transaction that resulted in the transfer of $1.04 USD from a source account to two destination accounts. The transfers show part of the payment going to the retailer and the other going to a tax account. 

{
   "@": "<http://example.com/transactions/1.A.F23BC>"
   "a": "<com:Transaction>",
   "com:transfer": [com:Transfer+]
   "com:amount": "1.04",
   "com:currency": "USD",
   "rdfs:comment": "Purchase of Bad Romance by Lady Gaga for $0.89 from Best Buy"
   "com:date": "2010-10-16T12:34:36.14532^^xsd:dateTime",
   "ps:originatingClient": "10.113.15.142^^iana:ipv4Address"
}

Deferred

This constant term is used to specify a deferred rate context. A deferred rate context specifies that the PayeeRule must be calculated after all non-deferred PayeeRules have been applied. This is most useful when a rule should be applied to the total cost of a good

Status
stable
Domain
com:rateContext

The following example demonstrates how a deferred rate context can be specified: 

{
   "a": "com:PayeeRule",
   "com:rate": "0.05",
   "com:rateType": "com:Cumulative",
   "com:currency": "USD",
   "com:destination": "",
   "rdfs:comment": "The explanation of the deferred rate goes here.",
   "com:rateContext": "com:Deferred"
}
